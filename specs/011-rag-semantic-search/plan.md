## Summary

The RAG & Semantic Search feature upgrades the current keyword-only vendor discovery. It implements an event-driven LLM embedding generation pipeline for vendors and events, stores vectors in Neon DB using `pgvector`, and provides semantic text similarity queries exposing a `@function_tool` to the AI agents. 

## Technical Context

**Language/Version**: Node.js ≥ 20 (Backend), Python ≥ 3.12 (AI Service)
**Primary Dependencies**: Fastify, Prisma, Pydantic, OpenAPI Agents SDK, LangChain (for chunking/retrieval only), Google GenAI/OpenAI (Embedding API)
**Storage**: Neon DB (PostgreSQL) with `pgvector` extension
**Testing**: `pytest-asyncio` + `respx` (AI Service)
**Target Platform**: Linux server
**Constraints**: LLM Embedding quota management; Zero LLM calls in test suite.
**Scale/Scope**: Impacts search capability completely.

## Phase 1: Database & pgvector Extension

**Context**: `VendorEmbedding` and `EventEmbedding` exist in `schema.prisma` with `Unsupported("vector(1536)")`, but we need to ensure the extension is enabled and create the actual vectors.

**Tasks**:
1. Execute `CREATE EXTENSION IF NOT EXISTS vector;` across all environments via explicit Prisma migration.
2. Confirm the `VendorEmbedding` and `EventEmbedding` tables are created properly with pgvector column types.

## Phase 2: Python Embedding Pipeline

**Context**: Embeddings should be generated by the AI service natively, not the Node backend.

**Tasks**:
1. Create `embeddings.py`: Integrates `langchain-google-genai` (or `openai`) context vectorization.
2. Ensure the LLM embedding provider client is initialized in FastAPI `lifespan` and passed via `Depends()`.
3. Read configured embedding model name & dimensions directly from Pydantic `BaseSettings`. No `os.environ`.

## Phase 3: Event-Driven Triggers

**Context**: The AI service should listen to the backend `EventBusService` (created in the Booking phase) for updates.

**Tasks**:
1. Create `semantic_indexer.py`.
2. Expose a webhook endpoint in FastAPI for the backend to push `vendor.updated`, `vendor.registered`, and `event.created` domain events.
3. Upon receiving an event, invoke `embeddings.py` to upsert the vector directly via `SQLModel` using async operations (`selectinload` for related data if necessary).

## Phase 4: Semantic Search

**Context**: Replacing point-based keyword scoring with cosine similarity.

**Tasks**:
1. Implement the DB retrieval logic in Python using the `pgvector` `<=>` (cosine distance) operator paired with SQL filter conditions (budget, category).
2. Add `semantic_search_vendors` definition in `vendor_tools.py`, including a strict docstring, taking natural language queries, embedding them, and searching the DB.
3. Fallback gracefully to standard text matching if LLM embedding invocation fails or returns zero results.
4. Replace `VendorDiscoveryAgent` score methods with a direct call to Semantic Search.

## Phase 5: Testing

**Tasks**:
1. Use `respx` to intercept embedding API calls in `test_vendor_discovery.py` — ensuring no live API calls are strictly enforced.
2. Verify embedding pipeline handles concurrent upserts smoothly.
