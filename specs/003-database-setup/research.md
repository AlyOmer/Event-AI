# Phase 0 Research: Database Setup

**Date**: 2026-04-07  
**Feature**: 003-database-setup  
**Architect**: Claude (Sonnet 4.6)

## Objective

Research cloud-only database setup for Event-AI monorepo using Neon Serverless PostgreSQL, with package-specific ORMs (Prisma for Backend, SQLModel for AI Service). Determine best practices for connection management, migrations, health monitoring, and developer onboarding.

## Key Findings

### Neon Serverless PostgreSQL

**What**: Neon is a serverless PostgreSQL platform with instant provisioning, branching, and pay-per-use billing. Connection pooling is built-in via `pgbouncer`.

**Connection String Formats**:
- Pooled: `postgresql://user:pass@host.pooler.neon.tech/db?connection_limit=20`
- Direct: `postgresql://user:pass@host.neon.tech/db` (for migrations)

**Branching**: Neon allows creating isolated database branches (like Git branches). Perfect for:
- Development: each developer has their own branch, seeded from template
- Staging: separate branch for QA
- Production: main branch

**Extensions**: pgvector must be enabled manually once per project (`CREATE EXTENSION vector;`). We can detect and fail with helpful message.

**Backup & PITR**: Neon provides automated backups with point-in-time recovery for last 7-30 days depending on plan. No custom backup scripts needed.

### Prisma Migrate (Backend Package)

**Tool**: Prisma Migrate (`prisma migrate`) is the migration system for Prisma ORM.

**Workflow**:
- Schema defined in `prisma/schema.prisma`
- `prisma migrate dev` - for development (with interactive prompt)
- `prisma migrate deploy` - for production (non-interactive, applies pending migrations)
- `prisma db push` - for prototyping (not for migrations; no down migrations)

**Migration Files**: Stored in `prisma/migrations/` with timestamps. Each migration has:
- `migration.sql` (up)
- `rollback.sql` (down) - **must be defined for reversibility**

**CLI Integration**: Use Prisma CLI via `npx prisma` or package binary. In CI/CD, use `prisma migrate deploy`.

**Best Practices**:
- Never use `db push` in production
- Always review generated migration SQL for unexpected destructive changes
- Use `--create-only` to generate migration without applying (then review before deployment)
- Down migrations must be provided; Prisma can sometimes auto-generate but should be reviewed

**Environment Variables**:
- `DATABASE_URL`: pooled connection (for application)
- `DIRECT_URL`: direct connection (for migrations) - helps with transactional DDL

**Connection Pooling**: Prisma uses its own connection pool. For serverless environments, careful pool sizing needed. Neon's pgbouncer also provides pooling.

### Alembic (SQLModel/SQLAlchemy)

**Tool**: Alembic is the migration tool for SQLAlchemy (and SQLModel).

**Workflow**:
- Models defined in Python using SQLModel classes
- `alembic revision --autogenerate -m "message"` generates migration script
- `alembic upgrade head` applies migrations
- `alembic downgrade -1` rolls back one
- Migration scripts stored in `alembic/versions/`

**Migration Scripts**: Python files with `upgrade()` and `downgrade()` functions. Must be hand-tuned after autogeneration to handle:
- Column renames (autogenerate may drop + recreate instead)
- Data migrations (transformations)
- Destructive operations flagged

**Async Support**: SQLAlchemy 2.0+ supports async. Use `asyncpg` driver. Engine created with `create_async_engine()`. Alembic can run async via `alembic` command with appropriate config, or use custom CLI wrapper.

**Environment**:
- Connection URL from env var (DATABASE_URL or DATABASE_URI)
- No separate DIRECT_URL needed; Alembic can use same URL but should use direct connection (asyncpg doesn't pool in the same way as Prisma)

**Best Practices**:
- Always review autogenerated migrations
- Test downgrades thoroughly
- Use batch operations for large data migrations to avoid locking
- Lock tables when necessary during destructive changes

**Configuration**: `alembic.ini` or programmatic configuration in `env.py`.

### Health Monitoring with PostgreSQL System Views

**pg_stat_activity**: Shows current connections and queries.
```sql
SELECT state, count(*) FROM pg_stat_activity GROUP BY state;
-- states: active, idle, idle in transaction, etc.
```

**pg_stat_statements**: Requires extension `pg_stat_statements` enabled. Provides query execution statistics.
```sql
SELECT query, calls, total_time, mean_time, rows FROM pg_stat_statements ORDER BY mean_time DESC LIMIT 10;
```
Shows slowest queries by average time.

**Connection Pool Metrics**: For PostgreSQL itself:
```sql
SELECT count(*) FROM pg_stat_activity; -- total connections
```
Application connection pool metrics (e.g., from node-postgres or asyncpg) are exposed directly via pool object.

**pgvector Extension**: Check with:
```sql
SELECT * FROM pg_extension WHERE extname = 'vector';
```
Enable: `CREATE EXTENSION IF NOT EXISTS vector;`

### Developer Onboarding in Cloud-Only Model

**Traditional Local DB**: Developers install PostgreSQL locally, run migrations, seed. Slow, error-prone, environment drift.

**Neon Branch Model**:
1. Developer signs up for Neon (or gets invited to project)
2. Creates a personal branch from template `main` branch (e.g., `dev/username`)
3. Gets connection string for that branch
4. Sets environment variables locally
5. Starts application → auto-migrates

**Advantages**:
- Instant setup (seconds)
- No local installation
- Full PostgreSQL features (extensions, etc.)
- Easy to reset: delete branch and recreate
- Production-like environment

**Challenges**:
- Shared project limits (concurrent connections, storage)
- Need to coordinate schema changes (everyone pulls migrations from git)
- Branch management: must create branch template (includes schema and seed data)

**Template Branch Setup**:
- Create `template` branch in Neon with initial schema
- Developers fork from `template`
- Script to automate: `neon branch create --name dev/username --parent template`

### Migration Linting

**Goal**: Catch missing down migrations and destructive changes before PR merge.

**Implementation**:
- For Prisma: inspect `prisma/migrations/*/rollback.sql` existence; parse for DROP operations
- For Alembic: check if `downgrade()` function is not `pass`; flag operations like `op.drop_column()`, `op.drop_table()`
- CI job runs lint; fails if violations found

**Destructive Change Detection**:
- Drop table
- Drop column
- Alter column type that may cause data truncation
- Requires `--confirm-destructive` flag in production deployment

### Seed Data Idempotency Patterns

**Problem**: Multiple developers run seeds concurrently on shared branch template or personal branches. Seeds must not fail on re-run.

**Solution**: Upserts with natural keys, and `seed_log` table.

**Seed Log Table**:
```sql
CREATE TABLE seed_log (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  seed_name VARCHAR(255) UNIQUE NOT NULL,
  applied_at TIMESTAMPTZ DEFAULT NOW(),
  checksum VARCHAR(255)
);
```
Each seed operation:
1. Record seed name in seed_log within same transaction as data upsert
2. If seed_name already exists, skip (or validate checksum if content changed)

**Implementation**: Both Prisma and Alembic can run custom scripts after migrations to execute seeds.

### Security: Secret Management

**Never commit secrets**: `.env` in `.gitignore`. Provide `.env.example` with required variables.

**Neon Credentials**: Use Neon's built-in connection string with limited permissions (read-write only to own database). Rotate via Neon dashboard.

**Application Secrets**: JWT secrets, API keys go in `.env` separately. Use `@lazyjson.com` or similar for production injection, not committed.

**Local Development**: Each developer sets their own Neon connection string in local `.env`.

### Performance Considerations

**Connection Pool Size**:Default should be conservative. Neon's free tier limits connections (max 20?). Configure pool size via `connection_limit` query param in DATABASE_URL.

**Async Operations**: All database access must use async drivers (`pg` for Node.js with async/await, `asyncpg` for Python). No sync blocking calls.

**Indexing**: Constitution requires indexes on foreign keys and commonly filtered columns. Use Prisma `@@index` and SQLModel/SQLAlchemy `Index` declarations. Monitor slow queries via pg_stat_statements.

**pgvector Indexes**: For semantic search tables with vector columns (1536 dimensions typically), use HNSW index for approximate nearest neighbor search:
```sql
CREATE INDEX ON items USING hnsw (embedding vector_cosine_ops);
```

### Observability

**Structured Logging**: Use Pino (Backend) and Python structlog (AI Service). Log migration operations, seed execution, connection failures.

**Metrics**: Custom metrics exposed via health endpoint; optionally integrate with Prometheus if infra exists.

**Tracing**: Not required for database layer at this stage; application-level spans will trace DB queries automatically via ORM instrumentation.

## Recommendations Summary

1. **Adopt cloud-only Neon model** – Simplifies onboarding, ensures parity
2. **Use package-specific ORMs** – Prisma (Backend), SQLModel (AI) per constitution
3. **Separate schema namespaces** – Prevent cross-package table conflicts
4. **Implement migration linting** – Enforce reversible migrations constitutionally
5. **Idempotent seeds with log table** – Safeguard against duplicate execution
6. **Health endpoints using pg_stat_views** – No external agents needed
7. **Document clear onboarding** – Quickstart guide with Neon branch creation
8. **Enforce async connections** – Constitution mandate; use async drivers
9. **Plan for pgvector** – Ensure extension enabled early
10. **Use environment variables for secrets** – Never commit credentials

## Open Questions (Resolved)

**Q**: Should we provision local PostgreSQL as fallback?  
**A**: No (clarified). Cloud-only. Developers who need offline work can use Neon local emulator or create local branch temporarily, but not officially supported.

**Q**: Do we need custom backup scripts?  
**A**: No. Neon provides automated backups with PITR. We document restore procedures.

**Q**: Who owns shared lookup tables (event types, vendor categories)?  
**A**: Backend package owns these core entities; AI Service references via foreign keys but does not manage.

**Q**: What about circular dependencies between packages?  
**A**: Use separate schemas; cross-package references use foreign keys to other schema's tables (qualified with schema name). Avoid bidirectional dependencies; use event-driven updates.
